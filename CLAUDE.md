## Product Description

**AIFoxx** (`aifox.com`) is a comprehensive AI tools directory with 1000+ tools, organized by category with compliance metadata (GDPR, SOC2, HIPAA, ISO27001), pricing tiers, and use cases. Users can browse, search, filter, and discover AI tools in one place.

**Current focus:** Adding a **Claude Skills** section — a directory of GitHub repos providing MCP servers and Claude Code integrations, linked to their relevant tools.

**Stack:**
- React 18 + React Router DOM 6 + Vite + TypeScript
- Tailwind CSS + Shadcn/ui (Radix primitives)
- Static JSON data (`src/data/tools.json` — 3.7MB, 1000+ tools)
- Fuse.js for client-side search, React Query for data state
- Deployed on Vercel

## Architecture

```
src/
├── App.tsx                  # Route definitions (React Router)
├── pages/                   # One file per route
│   ├── HomePage.tsx         # / — tool listing + search
│   ├── ToolDetailPage.tsx   # /ai/:slug — single tool
│   ├── CategoryPage.tsx     # /category/:category
│   ├── TagPage.tsx          # /tag/:tag
│   └── SkillsPage.tsx       # /skills — Claude skills (to be added)
├── components/
│   └── ui/                  # Shadcn/ui components (DO NOT edit these)
├── lib/
│   ├── tools.ts             # allTools, getToolBySlug(), getRelatedTools()
│   ├── skills.ts            # allSkills, getSkillsByToolSlug() (to be added)
│   └── search.ts            # Fuse.js search engine
├── types/
│   ├── tool.ts              # Tool type + Zod schema
│   └── skill.ts             # Skill type (to be added)
└── data/
    ├── tools.json           # Master tools data (DO NOT edit manually)
    ├── skills.json          # Claude skills data (generated by scraper)
    └── categoryColors.json  # Category → color mapping

scripts/
└── scrape-github-skills.mjs # GitHub API scraper → writes skills.json
```

**Key patterns:**
- All data is static JSON — no backend, no DB
- Pages import from `src/lib/` — never import `data/*.json` directly in components
- URL state managed via `useSearchParams()` for filters + pagination
- Shadcn components in `src/components/ui/` are never manually edited — use `npx shadcn@latest add` to update

## Versioning

**Branch strategy:**
- `main` — production (auto-deploys to Vercel)
- `dev` — integration branch, all feature branches merge here first
- `feature/<name>` — individual features (e.g., `feature/claude-skills`)
- `fix/<name>` — bug fixes
- `chore/<name>` — non-functional changes (deps, config, docs)

**Commit message format (Conventional Commits):**
```
feat: add Claude Skills page and scraper
fix: correct tool slug lookup for special characters
chore: update dependencies
docs: update CLAUDE.md architecture section
```

**Tagging:**
- Use semantic versioning: `v1.0.0`, `v1.1.0`, `v1.1.1`
- Tag on every production-ready release from `main`
- MAJOR: breaking changes to data schema or routing
- MINOR: new features (new page, new filter)
- PATCH: bug fixes, copy changes, style tweaks

## Security Guidelines

### Data & Privacy
- **No secrets in source code** — never commit API keys, tokens, or credentials
- Store GitHub API tokens in `.env.local` (already in `.gitignore`) or Vercel environment variables
- Never log user search queries or filter selections
- tools.json contains no PII — keep it that way

### Dependency Security
- Run `npm audit` before adding new dependencies
- Prefer well-maintained packages with >1M weekly downloads for core functionality
- Pin major versions in `package.json` — avoid `*` or `latest`
- Review `npm audit` output in every PR

### Input Handling
- All user inputs (search, filter params from URL) must be sanitized before use
- Validate URL params — do not trust `useSearchParams()` values without bounds checking
- Fuse.js search input: trim and limit to 200 chars max before passing to search engine
- Tool slugs from URL: validate against known slugs before rendering, return 404 otherwise

### Build & Deploy
- Never disable ESLint rules without a comment explaining why
- All PRs must pass `npm run build` cleanly — no TypeScript errors, no lint errors
- Do not commit to `main` directly — always go through a PR
- Vercel preview deployments are for review only — do not share preview URLs as "live" links

### What Claude Must Not Do
- Do not edit files in `src/components/ui/` — these are auto-generated Shadcn components
- Do not manually edit `src/data/tools.json` — use the scraper/validation scripts
- Do not add analytics tracking beyond the existing Vercel Analytics integration
- Do not introduce server-side code (this is a static SPA deployed on Vercel CDN)

## Testing & Code Review

### Before Every Commit
```bash
npm run lint          # ESLint — must pass clean
npm run build         # TypeScript compile + Vite build — must succeed
npm run validate      # Check tools.json for duplicate slugs + required fields
```

### Unit Tests
- Location: `src/__tests__/` or colocated `*.test.ts` files
- Framework: Vitest
- Run: `npm run test`
- Required for: all `src/lib/` functions (`tools.ts`, `skills.ts`, `search.ts`)
- Not required for: pure UI components, page components

### E2E Tests
- Framework: Playwright (`playwright.config.ts`)
- Run: `npm run test:e2e`
- Cover: homepage load, tool detail page, search working, `/skills` page (once built)

### Code Review Checklist
Before merging any PR:
- [ ] `npm run build` passes with zero errors
- [ ] No `any` types introduced without justification
- [ ] No hardcoded strings that should be constants
- [ ] No direct `data/*.json` imports in components (use `lib/` functions)
- [ ] New pages have `<title>` and `<meta description>` set
- [ ] New lib functions have at least one unit test
- [ ] Mobile layout checked (Tailwind responsive classes used correctly)
